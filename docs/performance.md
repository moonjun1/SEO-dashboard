# 성능 최적화

본 문서는 SEO Dashboard에 적용된 성능 최적화 사항을 정리합니다. 각 최적화의 문제 상황, 해결 방법, 측정 결과를 기술합니다.

---

## 1. 백엔드 성능 최적화

### 1.1 N+1 쿼리 해결

**문제**

대시보드 API 호출 시 Hibernate가 연관 엔티티를 건별로 조회하여 과도한 쿼리가 발생하였습니다.

| 엔드포인트 | 최적화 전 | 최적화 후 |
|-----------|----------|----------|
| 종합 대시보드 (`GET /api/v1/dashboard`) | 211개 쿼리 | 3개 쿼리 |
| 크롤링 결과 (`GET /crawl/jobs/{id}/results`) | 101개 쿼리 | 2개 쿼리 |

**해결 방법**

1. `default_batch_fetch_size` 설정 적용

```yaml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100
```

Hibernate가 Lazy Loading 시 `IN` 절을 사용하여 연관 엔티티를 일괄 조회하도록 변경하였습니다. 개별 `SELECT` 대신 `WHERE id IN (?, ?, ..., ?)` 형태의 배치 쿼리를 생성합니다.

2. 필요한 경우 `@EntityGraph` 또는 JPQL `JOIN FETCH`로 즉시 로딩 적용

**효과**: 대시보드 API 응답 시간이 대폭 단축되었으며, 데이터베이스 부하가 크게 감소하였습니다.

### 1.2 Redis 캐시 계층

**구조**

`@EnableCaching`과 Spring Cache Abstraction을 활용하여 엔드포인트별로 다른 TTL을 적용하였습니다.

| 캐시 대상 | TTL | 사유 |
|-----------|-----|------|
| 종합 대시보드 | 30초 | 실시간성이 중요하나 초당 다수 요청 발생 |
| 공개 SEO 분석 결과 | 5분 | 동일 URL 반복 분석 방지, 결과 변동 빈도 낮음 |
| SEO 점수 순위 | 2분 | 순위 변동은 분 단위로 충분 |

**캐시 매니저 구성**

용도별로 별도의 `RedisCacheManager`를 구성하여 TTL을 분리 관리합니다. 하나의 캐시 매니저에 모든 TTL을 혼합하지 않고, 캐시 이름별로 만료 정책을 설정합니다.

**캐시 무효화**

데이터 변경 시(크롤링 완료, 새 분석 결과 저장) 관련 캐시를 `@CacheEvict`로 즉시 무효화하여 오래된 데이터가 반환되는 문제를 방지합니다.

### 1.3 트랜잭션 배치 처리

**문제**

크롤링 시 페이지별로 개별 트랜잭션을 열어 INSERT하면 트랜잭션 오버헤드가 누적됩니다.

**해결: CrawlBatchPersister**

크롤링 결과를 10개 페이지 단위로 묶어서 하나의 트랜잭션에서 일괄 저장합니다.

```
페이지 1~10 크롤링 완료 → 배치 INSERT (1 트랜잭션)
페이지 11~20 크롤링 완료 → 배치 INSERT (1 트랜잭션)
...
```

**효과**: 트랜잭션 수 감소, DB 커넥션 점유 시간 단축, 크롤링 전체 소요 시간 개선

### 1.4 응답 크기 제한

크롤링 대상 웹페이지의 HTTP 응답 본문을 5MB로 제한합니다. 이는 비정상적으로 큰 페이지가 서버 메모리를 과도하게 사용하는 것을 방지하는 동시에, HTML 파싱 시간을 단축합니다.

### 1.5 open-in-view: false

```yaml
spring:
  jpa:
    open-in-view: false
```

Spring Boot의 기본값인 `open-in-view: true`는 HTTP 요청 전체 생명주기 동안 영속성 컨텍스트를 유지합니다. 이는 Controller에서 Lazy Loading이 발생할 수 있어 다음 문제를 일으킵니다.

- 예측 불가능한 쿼리 실행 (View 렌더링 중 N+1)
- DB 커넥션 장기 점유
- 성능 병목 추적 어려움

`false`로 설정하여 Service 계층에서만 DB 조회가 가능하도록 강제하고, 모든 필요한 데이터를 Service에서 DTO로 변환하여 반환합니다.

## 2. 크롤링 성능 최적화

### 2.1 병렬 크롤링 (Virtual Threads + Semaphore)

**문제**

기존 순차 크롤링은 페이지 하나의 응답을 기다린 후 다음 페이지를 요청하여 I/O 대기 시간이 누적되었습니다.

**해결**

Java 21의 Virtual Threads와 Semaphore를 결합하여 병렬 크롤링을 구현하였습니다.

```
[크롤링 큐]
     |
     v
[Virtual Thread Pool]
     |
     +→ Thread 1: 페이지 A 크롤링 (I/O 대기 중)
     +→ Thread 2: 페이지 B 크롤링 (I/O 대기 중)
     +→ Thread 3: 페이지 C 크롤링 (I/O 대기 중)
     +→ Thread 4: 페이지 D 크롤링 (응답 수신)
     +→ Thread 5: 페이지 E 크롤링 (파싱 중)
     |
  Semaphore (동시 요청 수 제한)
```

| 설정 | 값 | 목적 |
|------|-----|------|
| 동시 요청 수 | 5 (설정 가능) | 대상 서버 부하 제한 |
| 요청 타임아웃 | 10초 | 무응답 서버 대기 방지 |
| 요청 간 딜레이 | 200ms | 대상 서버 예의(politeness) 준수 |

**효과**

- 순차 처리 대비 3~5배 크롤링 속도 향상
- Virtual Threads는 OS 스레드를 점유하지 않아 수천 개의 동시 I/O가 가능
- Semaphore로 동시 요청 수를 제한하여 대상 서버에 과부하를 주지 않음

**Virtual Threads 선택 사유**

기존 Platform Thread는 생성/전환 비용이 높아 스레드 풀 크기가 제한적입니다. 크롤링은 대부분의 시간을 I/O 대기에 소비하므로, 경량 Virtual Threads가 적합합니다. 컨텍스트 스위칭 없이 수천 개의 동시 크롤링 작업을 효율적으로 처리할 수 있습니다.

## 3. 프론트엔드 성능 최적화

### 3.1 코드 스플리팅 (React.lazy)

**문제**

Recharts 라이브러리(342KB)가 초기 번들에 포함되어 모든 페이지의 로딩 시간에 영향을 주었습니다.

**해결**

`React.lazy`와 `Suspense`를 사용하여 차트가 포함된 페이지를 동적으로 로드합니다.

```
초기 로드: 핵심 UI (로그인, 대시보드 레이아웃)
필요 시 로드: 키워드 차트 페이지, AI 분석 결과 페이지
```

**효과**: 초기 번들 크기 감소, 첫 페이지 로딩 시간 단축

### 3.2 Error Boundary

코드 스플리팅으로 분리된 컴포넌트의 로딩 실패 시, 전체 애플리케이션이 중단되지 않도록 Error Boundary를 적용하였습니다. 네트워크 오류 시 사용자에게 재시도 안내를 표시합니다.

### 3.3 정적 데이터 외부화

`LandingPage`의 features 배열을 컴포넌트 외부로 이동하여 리렌더링 시 불필요한 배열 재생성을 방지하였습니다. React의 참조 비교(reference equality) 최적화가 정상 동작하도록 합니다.
